Maintaining Packages

Wasta-Linux: Maintain a Custom Package

This page is part of the Wasta-Linux customizing process.

Follow these steps to modify a wasta-custom-xyz package.  I will use "wasta-custom-ssg" as an example.

0. Setup (only needs to be done the first time)

  • Install needed dependencies:
    • build-essential
    • devscripts
    • dh-make
    • dpkg-dev
    • dput
  • In "Software Sources" application, "Other Software" tab, ensure you have the wasta-linux/wasta Source Code repository selected.
  • Make a folder somewhere in your personal home to hold the source code, something like this: "wasta-packages/wasta-custom-ssg/" (So, the "wasta-packages" could be in your Documents, for example: up to you!  You should have the actual package name as the final folder (so, "wasta-custom-ssg" for this example)
  • To automatically enter your name and email when building the packages, add following lines to ~/.profile (you will have to logout / login for these settings to take effect):

export DEBFULLNAME=Rik\ Shaw

export DEBEMAIL=email@address.com

  • NOTE the "\" delimiter to account for the space in the name (between "Rik" and "Shaw" in this example

1. Get Current Source

  • Right click on the wasta-custom-ssg folder (that you made above) in your file browser (Nemo), and choose "Open In Terminal."
  • Terminal command: "apt-get source wasta-custom-ssg" to download the current package source from Launchpad.
  • Back in Nemo, uncompress the downloaded file: it will create a ...wasta-packages/wasta-custom-ssg/wasta-custom-ssg-0.1.4 (matches version name) subfolder (this uncompressing may possibly happen automatically).

2. Create new version and make changes:

  • In a terminal, cd to the wasta-custom-ssg-0.1.4 folder (so you are in .../wasta-custom-ssg/wasta-custom-ssg-0.1.4).
  • Use "dch" to create the new version and enter comments regarding what changes you will make: Terminal command: "dch --newversion 0.1.5"
    • After saving, this will RENAME directory to match new version number (wasta-custom-ssg-0.1.5), so you may want to "cd .." to go up one level and then "cd wasta-custom-ssg-0.1.5" to get back to the working directory).
      • Remember "tab autocomplete" in the terminal.  Typing "cd wasta [tab]" should autocomplete to "cd wasta-custom-ssg-0.1.5", for example.
    • Later changelog additions to the same version can be made by just using the "dch" command.
  • Make any needed changes.  Reference to debian files:
    • debian/control: Most important debian file.  Defines dependencies and other package details.
    • debian/install: Defines files installed on the system when the package is installed.
      • Typically Wasta packages have an "install-files" folder where all the files installed by a package reside.  Confirm in this "debian/install" file.
      • Sometimes there is also a package name folder such as "wasta-custom-ssg" in this example, that is installed to "/usr/share/wasta-custom-ssg".  Again, this "debian/install" file will define these things.
    • debian/changelog: again, do NOT manually update but instead use "dch" (see above).
    • debian/postinst: Script executed after install of package. This should just call /usr/share/wasta-custom-ssg/wasta-custom-ssg-postinst.sh (I want this script easily accessible to others for reference: talk to me more if you want more gory details).

3. Build New Version

  • "debuild -b" to build .deb for local testing (creates the .deb file)
    • must have an OpenPGP key set up on the machine you do this from, that will sign the packages (creating a OpenPGP key is part of the Wasta-Linux Launchpad Setup)
    • Resulting .deb created "up" one directory (in the ...wasta-custom-ssg folder)
    • "Lintian" checks are done looking for any standards violations. Clean up any errors found, re-run. Some of the "warnings" can be ignored, as debuild -b makes different lintian checks than debuild -S seems to.  You can google the lintian phrase to see the warning description.
    • Use the created .deb to install manually on another local  machine for testing to ensure all changes are applied correctly.
      • The testing machine must have the same "architecture" (32 bit or 64 bit) as the machine you created the .deb on.
  • "debuild -S" to create source file package (required for Launchpad upload)
    • Again, resulting ".0.1.5_source.changes" file that would be uploaded to Launchpad is created "up" one directory (in the ...wasta-custom-ssg folder).

4. Upload new version to Launchpad

  • "dput ..." to upload to ppa
    • "dput ppa:wasta-linux/wasta  wasta-custom-ssg_0.1.5_source.changes"
      • This command is done from the base "wasta-custom-ssg" folder.  If you are still in the "wasta-custom-ssg-0.1.5" folder, then "cd .." to get up one directory.
      • Remember "tab" autocomplete in the terminal so you don't have to type this full thing!
    • Then, check build status: https://launchpad.net/~wasta-linux/+archive/wasta/+packages
